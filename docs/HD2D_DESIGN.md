# HD-2D 設計思想（黒猫飯店 探索画面）

オクトパストラベラー／ポケモン ピクセルリメイク風の「HD-2D」を本作で再現するための
画づくりの考え方と、本プロジェクト（Godot 4.6・GL Compatibility・モバイル縦・既存ドット絵キャラ）
での採用方針をまとめる。実装の具体は `docs/HD2D.md`、素材調達は `docs/HD2D_ASSETS.md` を参照。

## 1. HD-2D とは

**「3D 空間の中に、2D ドット絵のキャラクターを立たせる」** 表現。
背景・地形は 3D（奥行き・ライティング・影・被写界深度を持つ）だが、
キャラクターは 2D スプライトをビルボードで立てる。
3D の情報量・光の表現と、2D ドット絵の親しみやすさ・記号性を両立させるのが狙い。

---

## 2. 画づくりの原則

note 記事の結論とオクトラの定番レシピを統合すると、要点は次の 5 つ。

### 2.1 背景は「3D ×荒い解像度テクスチャ」
- 高精細なジオメトリに**あえて低解像度の“荒い”テクスチャ**を貼ると、ドット絵キャラと馴染む。
- 記事の到達点：**voxel（3D ドット）フィールド**でもこの質感は出せる。
  自作の高精細モデルは 1 人開発には重すぎるため、**voxel / ローポリ × 荒テクスチャ**が現実解。
- Godot では地形/モデルの `texture_filter` を **Nearest** にすると、この「荒さ」をそのまま活かせる。

### 2.2 キャラは「2D ドット絵ビルボード」
- スプライトを `Sprite3D` の **Y 固定ビルボード**で立てる（直立のままカメラを向く）。
- Nearest フィルタ＋アルファスシザーで輪郭をくっきりさせ、影を落とす（`docs/HD2D.md` 参照）。
- 本作は既存のピクセルキャラ（`assets/generated/sprites/`）をそのまま流用できる＝**作り直し不要**。

### 2.3 明るい照明と強いコントラスト
- 記事が「最もオクトラっぽくなった」要因に挙げたのが **照明を明るく・コントラストを強める**こと。
- 平坦に見せないため、**影（DirectionalLight3D のシャドウ）**を必ず入れて立体感を出す。

### 2.4 奥行き（レイヤー感）
- 手前・中景・遠景の重なりが HD-2D の生命線。**奥にオブジェクトを置き、視差と前後関係**を作る。
- 傾けた perspective カメラ（見下ろし）で奥行きを強調する。

### 2.5 仕上げのポストプロセス
オクトラの“あの空気感”はポスト処理で決まる。優先度順：
1. **ブルーム（glow）** … 光源・ハイライトをにじませる。HD-2D の柔らかさの核。
2. **被写界深度（DOF / tilt-shift）** … 手前/遠景をぼかしてミニチュア感。**効果は大きいが Forward+ 専用**。
3. **ヴィネット** … 画面端を暗く落として中央へ視線を集める。レンダラ非依存。
4. **光芒・パーティクル（埃・花びら・ホタル）** … 空気の密度。素材で足すのが本作の方針。

## 3. 本プロジェクトでの方針

| 原則 | 本作での採用 | 備考 |
|---|---|---|
| 3D 背景 ×荒テクスチャ | **フリーの voxel / ローポリ素材**を調達（自作しない） | `docs/HD2D_ASSETS.md`。Godot は glTF/GLB ネイティブ対応 |
| 2D ドット絵ビルボード | 既存スプライトを `Sprite3D` で流用 | 実装済みプロトタイプあり |
| 明るい照明＋影 | DirectionalLight3D（影 ON・shadow_blur で柔らか） | GL Compatibility で動作 |
| 奥行き | 傾けた perspective ＋ 周回カメラ | Q/E 回転・R/F ズーム |
| ブルーム | `Environment.glow`（bloom 控えめ） | GL Compatibility 対応 |
| ヴィネット | 全画面シェーダ | 実装済み |
| **被写界深度（DOF）** | **見送り（要 Forward+）** | 採用するならレンダラ切替が前提（§4） |
| 光芒・パーティクル | **自前実装せずフリー素材で足す** | 方針転換。エフェクト素材を調達 |

### 重要な制約
- 本プロジェクトは **GL Compatibility レンダラ**（モバイル/Web 配信のため）。
  この上では **DOF・SSIL・SSAO・ボリュメトリックライトは出ない**。
  ブルーム（glow）・影・ヴィネットは動く。
- 縦画面（720×1280）モバイル UI。3D は `SubViewport` に内包し、既存の Control UI と共存させる。
- 既存の探索画面 `src/ui/dive_view.gd`（2D `_draw`）は壊さない。HD-2D は別画面として段階導入する。

## 4. レンダラ選択のトレードオフ

| レンダラ | ブルーム | DOF/SSIL/SSAO | モバイル/Web 負荷 | 判断 |
|---|---|---|---|---|
| **GL Compatibility**（現状） | ○ | ✕ | 軽い | コアはこのまま。DOF は諦める |
| **Mobile** | ○ | △（一部） | 中 | 中間案。要検証 |
| **Forward+** | ◎ | ◎ | 重い | DOF を本気で出すなら。配信負荷とトレードオフ |

DOF（tilt-shift）はオクトラ感の決め手だが、本作のモバイル/Web 配信方針とは相性が悪い。
**まずは GL Compatibility のまま、照明・コントラスト・奥行き・素材の質で寄せる**のが現実的。
DOF が欲しくなった段階で Forward+ 版を別ビルドで検証する。

## 5. 実装ロードマップ

1. **プロトタイプ評価**（済）：`hd2d_test.tscn` で見下ろし・ビルボード・回転カメラ・ヴィネットを確認。
2. **素材差し替え**：プレースホルダのタイル地面/ボックスを、フリー voxel/ローポリ素材に置換（§素材カタログ）。
3. **質感調整**：素材の `texture_filter=Nearest`・ライト強度・コントラスト・glow を詰める。
4. **本採用の判断**：「店周辺を歩く新画面」として段階導入（コアループは温存）。
5. **任意**：Forward+ 版で DOF を試し、効果と負荷を比較。
